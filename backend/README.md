# Todo API Backend

FastAPI backend for the Todo application with JWT authentication and user isolation.

## Architecture

- **Framework**: FastAPI (async/await)
- **Database**: Neon PostgreSQL with asyncpg driver
- **ORM**: SQLModel (Pydantic + SQLAlchemy)
- **Authentication**: JWT tokens via Better Auth
- **Security**: User isolation enforced at database query level

## Setup

### 1. Install Dependencies

```bash
cd backend
pip install -r requirements.txt
```

### 2. Configure Environment

Copy `.env.example` to `.env` and configure:

```bash
# CRITICAL: Must match the secret in root .env
BETTER_AUTH_SECRET=your-256-bit-secret-here

# Neon PostgreSQL connection string
DATABASE_URL=postgresql://user:password@ep-xxx.us-east-2.aws.neon.tech/neondb

# API Configuration
API_PORT=8000
API_HOST=0.0.0.0
CORS_ORIGINS=http://localhost:3000,http://127.0.0.1:3000

# Logging
LOG_LEVEL=INFO
ENVIRONMENT=development
```

**Generate a secure secret:**
```bash
openssl rand -base64 32
```

### 3. Initialize Database

The database tables are automatically created on first startup. To manually verify:

```bash
cd backend
python -c "from src.db import init_database; import asyncio; asyncio.run(init_database())"
```

## API Endpoints

All endpoints (except `/health` and `/`) require JWT authentication via `Authorization: Bearer <token>` header.

### Public Endpoints

- `GET /` - API information
- `GET /health` - Health check (database connectivity)

### Protected Endpoints (require JWT)

- `POST /api/todos` - Create a new todo
- `GET /api/todos` - List all todos for authenticated user
- `GET /api/todos/{id}` - Get a specific todo
- `PUT /api/todos/{id}` - Update a todo
- `DELETE /api/todos/{id}` - Delete a todo
- `PATCH /api/todos/{id}/complete` - Toggle todo completion status

## Authentication

### JWT Token Structure

Tokens are generated by Better Auth (Next.js frontend) and verified by FastAPI:

```json
{
  "sub": "user_123abc",           // User ID (extracted for queries)
  "email": "user@example.com",    // User email
  "iat": 1704672000,              // Issued at timestamp
  "exp": 1705276800,              // Expiration timestamp
  "iss": "todoweb",               // Issuer (validated)
  "aud": "todoweb-api"            // Audience (validated)
}
```

### Security Features

1. **JWT Verification**: All tokens verified with `BETTER_AUTH_SECRET`
2. **User Isolation**: All database queries filter by `user_id` from JWT
3. **Information Hiding**: Returns 404 (not 403) when accessing other users' todos
4. **CORS Protection**: Only allows requests from configured origins
5. **Async Operations**: Non-blocking I/O for better performance

## Database Schema

### `todos` Table

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| id | UUID | PRIMARY KEY | Unique identifier |
| user_id | VARCHAR | NOT NULL, INDEXED | Owner user ID from JWT |
| title | VARCHAR(200) | NOT NULL | Todo title |
| description | VARCHAR(2000) | NULLABLE | Optional description |
| is_completed | BOOLEAN | NOT NULL, DEFAULT false | Completion status |
| created_at | TIMESTAMP | NOT NULL | Creation timestamp (UTC) |
| updated_at | TIMESTAMP | NOT NULL | Last update timestamp (UTC) |

**Indexes:**
- Primary: `id` (UUID)
- Secondary: `user_id` (for user isolation queries)
- Composite: `(user_id, is_completed)` (for filtered lists)

## Error Handling

### HTTP Status Codes

- `200 OK` - Successful GET/PUT/PATCH
- `201 Created` - Successful POST
- `204 No Content` - Successful DELETE
- `400 Bad Request` - Invalid request format
- `401 Unauthorized` - Invalid/missing/expired JWT token
- `404 Not Found` - Todo not found or doesn't belong to user
- `422 Unprocessable Entity` - Validation error
- `500 Internal Server Error` - Server error
- `503 Service Unavailable` - Database connection failed

## Security Considerations

### Production Deployment

1. **HTTPS Only**: Always use HTTPS in production
2. **Secret Rotation**: Change `BETTER_AUTH_SECRET` requires re-authentication
3. **Environment Variables**: Never commit `.env` file
4. **Database Credentials**: Use connection pooling and secure credentials
5. **Rate Limiting**: Implement rate limiting for production
6. **Monitoring**: Set up logging and monitoring for security events

### User Isolation

All database queries MUST filter by `user_id`:

```python
# CORRECT: Filters by user_id from JWT
statement = select(Todo).where(
    col(Todo.id) == todo_id,
    col(Todo.user_id) == user_id  # CRITICAL
)

# WRONG: No user_id filter (security vulnerability)
statement = select(Todo).where(col(Todo.id) == todo_id)
```

## Performance

### Optimization Tips

1. **Connection Pooling**: Configured for Neon serverless (pool_size=5)
2. **Async Operations**: All I/O operations use async/await
3. **Database Indexes**: Composite index on `(user_id, is_completed)`
4. **Query Optimization**: Use `.where()` filters before `.all()`

### Monitoring

- Health check endpoint: `/health`
- Database connectivity: Verified on each health check
- Response times: Monitor p95 latency (target: <200ms)
